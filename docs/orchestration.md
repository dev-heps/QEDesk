# QEDesk Token-Efficient Orchestration

This document describes the orchestration layer that is actually present in
this repository, then separates it from research ideas that are still roadmap
items.

QEDesk's rule is unchanged: AI audits and proposes. Lean 4 verifies.

## Repository Grounding

As of this revision, orchestration is a wired prototype, not a fully autonomous
proof agent.

Implemented in the repository:

| Capability | Status | Repository path |
| --- | --- | --- |
| CLI entrypoint | wired | `bin/qedesk`, `bin/qedesk.bat` |
| OpenRouter router | wired-prototype | `tools/qedesk_router.py`, `qedesk-router.json` |
| Cost ledger | wired-prototype | `tools/qedesk_ledger.py`, `build/qedesk-ledger.sqlite` |
| QEDesk node parser | wired | `tools/qedesk_sync.py` |
| Lean declaration extraction | wired | `tools/qedesk_sync.py` |
| Lean Blueprint source generation | wired | `blueprint/src/content.tex`, generated by `tools/qedesk_sync.py` |

| Lean Blueprint web output | wired | `./bin/qedesk blueprint`, `blueprint/web/` |

Not implemented yet:

| Capability | Current status |
| --- | --- |
| Automatic theorem-search RAG | Roadmap. `LeanSearchClient` is available through mathlib, but QEDesk does not yet expose a search command. |
| Loogle/local theorem index | Roadmap. Not installed or wired. |

| Learned uncertainty calibration | Roadmap. `tau_uncertainty` exists in config, but there is no calibration routine yet. |
| Strict JSON schema validation for model responses | Roadmap. The router asks for compact JSON and parses it, but it does not yet validate each response against a schema file. |
| Automatic patching of Lean or TeX from audit JSON | Roadmap. Router output is advisory only. |
| Full custom Blueprint UI | Roadmap. QEDesk currently uses upstream Lean Blueprint HTML and injects a small navigation switcher. |

The canonical status source for third-party tools is
`docs/INTEGRATIONS.md`. This file must not imply that a research project is
integrated when `docs/INTEGRATIONS.md` marks it as roadmap or not-integrated.

## Actual Data Flow

The current QEDesk loop is file-based and local.

```text
src/main.tex
  QEDesk comment markers
  natural-language proof notes
        |
        v
tools/qedesk_sync.py
        |
        +--> generated Lean map in src/main.tex
        +--> blueprint/src/content.tex
        +--> build/qedesk-dag.json
        +--> build/qedesk-dag.mmd

src/Proof.lean
  Lean declarations and proof candidates
        |
        v
./bin/qedesk lean
  Lean/Lake verification
```

The model-routing path is separate:

```text
src/main.tex + src/Proof.lean
        |
        v
tools/qedesk_router.py
        |
        +--> compact JSON prompt
        +--> OpenRouter model tier
        +--> build/qedesk-audit.json
        +--> build/qedesk-formalize.json
        +--> build/qedesk-repair.json
        +--> build/qedesk-ledger.sqlite
```

The router does not verify proofs. It only produces local audits,
formalization skeletons, and repair hints.

## Implemented Commands

The orchestration commands exposed through `bin/qedesk` are:

```bash
./bin/qedesk audit src/main.tex
./bin/qedesk audit --dry-run --max-nodes 1 src/main.tex
./bin/qedesk formalize src/main.tex --node <node_id>
./bin/qedesk repair src/main.tex --node <node_id> --lean-error "..."
./bin/qedesk repair src/main.tex --node <node_id> --error-file build/lean-error.txt
./bin/qedesk route <node_id>
./bin/qedesk cost
./bin/qedesk init-ledger
```

Use `--dry-run` before spending OpenRouter credits. A dry run prints the exact
messages that would be sent and does not append to the ledger.

The router can also operate on worksheet directories:

```bash
./bin/qedesk audit --dry-run worksheets/<slug>
./bin/qedesk formalize worksheets/<slug> --node <node_id>
./bin/qedesk repair worksheets/<slug> --node <node_id> --lean-error "..."
```

## Router Implementation

`tools/qedesk_router.py` implements three stages:

| Stage | Output file | Purpose |
| --- | --- | --- |
| `audit` | `build/qedesk-audit.json` | Classify local proof gaps and produce study hints. |
| `formalize` | `build/qedesk-formalize.json` | Suggest Lean statements or small skeletons for one node. |
| `repair` | `build/qedesk-repair.json` | Diagnose a Lean/local error and suggest the next small repair direction. |

The router builds prompts from:

1. the current node;
2. up to three parent nodes from `% QEDesk[uses=...]`;
3. the stage schema embedded in `tools/qedesk_router.py`;
4. optional Lean error text for repair;
5. remaining worksheet token budget from `tools/qedesk_ledger.py`.

It clips node bodies and Lean errors using an approximate character-based token
budget. It does not use a provider tokenizer yet.

## Model Configuration

The Python router reads `qedesk-router.json`.

Current tiers:

| Tier | Model ID | Role |
| --- | --- | --- |
| `flash` | `deepseek/deepseek-v4-flash` | Cheap first-pass audit, skeleton, and repair hints. |
| `pro` | `deepseek/deepseek-v4-pro` | Escalation after repeated low-tier failures. |
| `critic` | `deepseek/deepseek-r1` | Deep critic for conceptual or statement-level errors. |

Current budget values:

| Config key | Value |
| --- | ---: |
| `budget.worksheet_tokens` | 12000 |
| `budget.node_input_tokens` | 384 |
| `budget.node_output_tokens` | 256 |
| `budget.lean_error_tokens` | 320 |
| `routing.retries.flash` | 2 |
| `routing.retries.pro` | 1 |
| `routing.retries.critic` | 0 |

Current critic-triggering error types:

```json
[
  "hidden_premise",
  "quantifier_drift",
  "counterexample",
  "statement_drift"
]
```

OpenCode has a related but separate configuration in `opencode.json`:

| OpenCode agent | Model | Purpose |
| --- | --- | --- |
| `qedesk-tutor` | `openrouter/deepseek/deepseek-v4-pro` | Main proof-audit tutor. |
| `qedesk-fast` | `openrouter/deepseek/deepseek-v4-flash` | Low-cost local explorer. |
| `qedesk-verifier` | `openrouter/deepseek/deepseek-r1-0528` | Strict re-auditor for subtle proof gaps. |

The OpenCode agents are human-facing assistants. The Python router is a CLI
workflow. They share the same philosophy, but they are not the same runtime.

## Actual Routing Algorithm

The current implementation is intentionally simple.

```text
route_tier(config, ledger, worksheet_id, node_id, error_type):
  if error_type is in routing.critic_error_types:
    return critic

  if flash failures for this worksheet/node are below retries.flash:
    return flash

  if pro failures for this worksheet/node are below retries.pro:
    return pro

  return critic
```

Important consequences:

1. `tau_uncertainty` is currently a configuration placeholder.
2. No learned uncertainty model is trained yet.
3. Success or failure means router-call outcome, not Lean proof status.
4. Lean status is stored in the ledger schema, but the current router does not
   automatically update it from a Lean build.

This is enough for cautious cost tracking and dry-run inspection. It is not
yet a research-grade adaptive cascade.

## Ledger Implementation

`tools/qedesk_ledger.py` stores calls in SQLite.

Current columns:

| Column | Meaning |
| --- | --- |
| `created_at` | UTC timestamp. |
| `worksheet_id` | Usually the TeX path or worksheet path. |
| `node_id` | QEDesk node ID. |
| `stage` | `audit`, `formalize`, or `repair`. |
| `tier` | `flash`, `pro`, or `critic`. |
| `model_name` | Provider model ID. |
| `prompt_tokens` | Provider usage if available, otherwise local estimate. |
| `completion_tokens` | Provider usage if available, otherwise local estimate. |
| `estimated_cost_usd` | Calculated from `qedesk-router.json` model rates. |
| `elapsed_ms` | End-to-end call latency. |
| `outcome` | `ok` if JSON parsing succeeded; `fail` otherwise. |
| `lean_status` | Defaults to `unchecked`. |
| `error_type` | Optional local error classification. |
| `tags_json` | Extra metadata such as exception type or content length. |

`./bin/qedesk cost` reports:

1. number of calls;
2. prompt tokens;
3. completion tokens;
4. estimated cost;
5. cost per distinct node touched by the router;
6. model-level subtotals.

The current cost summary is a spend ledger, not a proof-verification report.
For verified proof status, use `./bin/qedesk lean` and inspect Lean output.

## Prompt Shape

The router sends two chat messages.

System message:

```text
You are QEDesk's proof-audit router. AI is an auditor and proposer;
Lean 4 is the final verifier. Return compact JSON only. Never provide
a full finished proof unless explicitly asked by the human outside this
tool. Prefer local hints, typed assumptions, and small subgoals.
```

User payload contains:

```json
{
  "instruction": "stage-specific instruction",
  "policy": [
    "Do not use chain-of-thought prose.",
    "Do not invent unavailable Lean library names.",
    "If the statement is ambiguous, say what must be clarified.",
    "Keep the answer short enough for the remaining budget."
  ],
  "payload": {
    "stage": "audit | formalize | repair",
    "worksheet_id": "...",
    "node": {
      "id": "...",
      "kind": "...",
      "title": "...",
      "lean_decl": "...",
      "status": "...",
      "uses": ["..."],
      "span": {"start_line": 1, "end_line": 10},
      "body": "..."
    },
    "parents": [],
    "lean_error": "...",
    "error_type": "...",
    "remaining_token_budget_for_worksheet": 12000,
    "response_schema": {}
  }
}
```

The model response is parsed as JSON. If the model returns text around JSON,
the parser tries to extract the outer object. If parsing fails, the raw text is
stored with `parse_error`.

## Blueprint Implementation

`tools/qedesk_sync.py` is the current bridge from notes and Lean declarations
to visual proof structure.

Implemented:

1. Parses QEDesk comment markers in LaTeX:

   ```tex
   % QEDesk[node=mem_power_set][kind=lemma][title=Membership in the power set][lean=mem_power_set][status=checked]
   % QEDesk[uses=def_power_set]
   ```

2. Extracts Lean declarations from `src/Proof.lean`.
3. Marks a Lean declaration as checked when its block does not contain
   `sorry`.
4. Generates Lean Blueprint-compatible `content.tex`.
5. Generates `build/qedesk-dag.json`.
6. Generates `build/qedesk-dag.mmd`.
7. Renders a small set of Lean statement patterns as TeX and falls back to a
   code rendering for unknown statements.

Not implemented:

1. No full LaTeX AST parser.
2. No automatic inference of a proof graph from arbitrary prose.
3. No bidirectional sync from Blueprint HTML back into TeX.
4. No custom graph editor.

`./bin/qedesk blueprint` runs upstream `leanblueprint web`, then
`tools/qedesk_blueprint_ui.py` injects a small switcher between:

```text
blueprint/web/index.html
blueprint/web/dep_graph_document.html
```

To view the result:

```bash
./bin/qedesk blueprint
./bin/qedesk serve
```

Then open:

```text
http://localhost:8000/
http://localhost:8000/dep_graph_document.html
```

## Research Synthesis

The following papers are used as design references only. They do not imply
that QEDesk contains those systems.

References were checked against their canonical arXiv pages on 2026-07-07.
Some are not Lean-specific; they are included only when the engineering idea
transfers to a budgeted proof-audit loop.

| Reference | What QEDesk uses from it | Current implementation status |
| --- | --- | --- |
| [Goedel-Architect][goedel-architect] | Use a blueprint graph before local proof refinement. | Partially implemented through QEDesk markers, Lean Blueprint generation, and DAG sidecars. No automatic blueprint-generation agent yet. |
| [EconProver][econprover] | Avoid expensive reasoning on every attempt. Escalate only after local evidence. | Partially implemented as Flash -> Pro -> critic routing by failure count and error type. No uncertainty model yet. |
| [Evaluation of LLMs for Mathematical Formalization in Lean][lean-formalization-eval] | Lean feedback is valuable; refine from errors instead of resampling blindly. | Implemented manually through `repair --lean-error` / `--error-file`. No automatic MCP error capture yet. |
| [FrugalGPT][frugalgpt] | Use model cascades for cost control. | Implemented as a three-tier OpenRouter config. |
| [Dynamic Model Routing and Cascading][dynamic-routing] | Keep routing logic outside prompts. | Implemented in `route_tier`, but with simple heuristics. |
| [UCCI][ucci] | Calibrate escalation thresholds from local outcomes. | Roadmap. `tau_uncertainty` is config only. |
| [SelfBudgeter][selfbudgeter] | Expose remaining budget and constrain output length. | Partially implemented through `remaining_token_budget_for_worksheet` and `max_tokens`. |
| [Adaptive Test-Time Compute Allocation][adaptive-compute] | Spend should be bounded at worksheet level, not just per call. | Partially implemented through `worksheet_tokens` and ledger accounting. No optimizer yet. |

## Why These References Matter

QEDesk is not trying to select one perfect model. The repository is moving
toward a replaceable workflow:

```text
student proof
-> local audit
-> explicit blueprint nodes
-> one Lean target
-> Lean check
-> small repair hint
-> ledger entry
-> repeat only where needed
```

This makes the model tier replaceable. If OpenRouter prices or model quality
change, the workflow should continue to work by editing `qedesk-router.json`
and `opencode.json`, not by rewriting the proof-audit protocol.

## Roadmap

Near-term engineering work that fits the existing repository:

1. Add JSON schema files for `audit`, `formalize`, and `repair` responses.
2. Rename the ledger metric from "checked node" to "touched node", or update
   it from real Lean status.
3. Add a command that captures the latest Lean error into a file and passes it
   to `qedesk repair`.
4. Add a theorem-search command around LeanSearchClient or a local search
   backend.
5. Add a calibration command that reads ledger outcomes and updates routing
   thresholds.
6. Add an optional audit-to-marker workflow that proposes QEDesk comments but
   requires human approval before editing TeX.
7. Keep full automatic proof writing out of the default student workflow.
